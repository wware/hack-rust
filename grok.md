# Grok

`bohemia_graph` is a Rust knowledge-graph engine over a named-entity-recognition dataset derived from "A Scandal in Bohemia" by Arthur Conan Doyle. At build time, `build.rs` reads four JSONL files (entities, events, moments, triplets) with `serde_json`, and emits a generated `bohemia_data.rs` containing four `pub static` arrays of structs with `&'static str` / `&'static [u32]` fields — the data is baked into the binary, requiring no runtime file I/O. At startup, `loader.rs` converts those statics into owned `Vec<Node>`, which `graph.rs` indexes into two `HashMap`s (`by_id`, `out_edges`/`in_edges`) for O(1) node lookup and edge traversal. The library is compiled as a `cdylib` exposing two C-ABI families: a JSON-string family (`ffi.rs`) that any C-compatible caller can use, and a native-SCM family (`ffi_scm.rs`, x86_64 macOS only) that returns values already built with libguile's C API so Guile Scheme receives real alists/lists with no JSON round-trip. Python callers use either a zero-dependency `ctypes` wrapper (`bohemia_graph.py`) that JSON-decodes each response, or an optional PyO3 extension (`python-native` feature) that skips the JSON layer entirely.

## Types

There are no Pydantic models — all structured data is owned by Rust. On the Python side, query results are plain `dict` / `list[dict]` values decoded from JSON. The Rust type layer has two concerns: the graph node model and a family of compile-time static record types used by the generated data module.

### Graph node types · `src/types.rs`

`Node` is a tagged enum (`entity` / `statement`) that is the unit of storage in the graph. `EntityNode` represents any person, place, event, or moment; `StatementNode` is a typed, directed, provenanced assertion (subject → predicate → object) that is itself a first-class node and can be the subject or object of other statements. `TruthStatus` is a five-variant enum (`asserted_true`, `asserted_false`, `hypothetical`, `disputed`, `retracted`) stored on every `StatementNode`. `EdgeFilter` carries optional predicate and truth-status constraints for edge queries.

| Type | Role |
|---|---|
| `Node` | Tagged enum; the unit of graph storage |
| `EntityNode` | Person / location / object / event / moment |
| `StatementNode` | Typed directed assertion with provenance |
| `TruthStatus` | Epistemic status of a statement |
| `EdgeFilter` | Optional filters passed to edge-query functions |

### Static record types · `src/types.rs`

Four parallel `Static*Record` structs use `&'static str` / `&'static [u32]` / `&'static [&'static str]` lifetimes so they can live in `static` arrays generated by `build.rs`. They mirror the JSONL schemas exactly; `StaticTripletRecord.truth_status` is a `TruthStatus` enum value, not a string.

| Type | JSONL source |
|---|---|
| `StaticEntityRecord` | `bohemia_entities.jsonl` |
| `StaticEventRecord` | `bohemia_events.jsonl` |
| `StaticMomentRecord` | `bohemia_moments.jsonl` |
| `StaticTripletRecord` | `bohemia_triplets.jsonl` |

> **Note:** `StaticTripletRecord.extraction_confidence` is `f64`. Rust allows `f64` in `static` initializers; the build script emits values with `{:?}` formatting so they always include a decimal point and round-trip exactly.

## How it fits together

`build.rs` runs at compile time, reads the four JSONL files from `CARGO_MANIFEST_DIR`, and writes `$OUT_DIR/bohemia_data.rs`; `src/data.rs` pulls that file in with `include!`. At program start, `loader::load_graph_data(sentence_cutoff)` iterates the four static arrays and produces a `Vec<Node>`, which `Graph::new` indexes into `by_id` (node lookup), `out_edges` (subject → statement IDs), and `in_edges` (object → statement IDs). The two FFI layers (`ffi.rs` for JSON, `ffi_scm.rs` for SCM) wrap the same `Graph` behind an opaque C pointer; Python reaches it via `ctypes` and JSON decoding while Guile receives native SCM values built directly in Rust using libguile's C API.

## Source Files

Eight substantive Rust modules plus two Python files. The Rust modules divide cleanly into: data pipeline (build.rs → data.rs → loader.rs), graph core (types.rs, graph.rs), and FFI surfaces (ffi.rs, ffi_scm.rs, guile_sys.rs). The Python layer is a two-level wrapper (ctypes shim → high-level class) with an optional PyO3 alternative.

### `build.rs`

Runs at compile time via Cargo's build-script mechanism. Emits `cargo:rerun-if-changed` directives for the four JSONL files, then reads each with `serde_json::Value` (no struct deserialization), escapes strings for Rust string literals, and writes `$OUT_DIR/bohemia_data.rs` containing `ENTITY_RECORDS`, `EVENT_RECORDS`, `MOMENT_RECORDS`, `TRIPLET_RECORDS` as `&[Static*Record]` statics. Also emits `rustc-link-lib` and `rustc-link-search` directives to link libguile-3.0 when the target is `x86_64-apple-darwin`. Has no dependencies beyond `serde_json` (listed under `[build-dependencies]`).

### `src/data.rs`

Trivial bridge: imports the four `Static*Record` types and `TruthStatus` into scope, then `include!`s the generated file. This is the only file that references `env!("OUT_DIR")`.

### `src/loader.rs`

Single public function `load_graph_data(sentence_cutoff: Option<u32>) -> LoadedData`. Iterates `ENTITY_RECORDS`, `EVENT_RECORDS`, and `MOMENT_RECORDS` converting `&'static str` fields to owned `String`s (events and moments become `EntityNode`s with `kind = "event"` / `"moment"`). Iterates `TRIPLET_RECORDS`, applying the optional sentence-ID cutoff filter, and converts each to `StatementNode`. Returns `LoadedData { nodes, warnings }`. No file I/O; no serde. Depends on `crate::data` and `crate::types`.

### `src/graph.rs`

`Graph` struct with three `HashMap`s: `by_id: HashMap<String, Node>`, `out_edges: HashMap<String, Vec<String>>` (entity ID → statement IDs where that entity is subject), `in_edges: HashMap<String, Vec<String>>` (entity ID → statement IDs where that entity is object). `Graph::new` populates all three in a single pass. Public methods: `get(id)`, `describe(id)`, `edges_from(id, filter)`, `edges_to(id, filter)`, `bfs(seeds, hops, truth_values)`, `transitive_closure(start, predicate)`. `canonicalize_id` converts `https://bakerstreet.fandom.com/wiki/X` → `wiki:X` and is also exported for FFI callers. BFS defaults to `asserted_true`-only traversal when no truth filter is supplied.

### `src/ffi.rs`

C-ABI exports usable from any language with a C FFI. `OpaqueGraph` is a public newtype wrapper around `Graph` so C callers hold an opaque pointer. Lifecycle: `graph_new()` → `graph_load(handle, …paths…, sentence_cutoff)` → `graph_destroy(handle)`. Path arguments to `graph_load` are accepted but ignored (data is embedded); only `sentence_cutoff` has effect. Query functions (`graph_get`, `graph_describe`, `graph_edges_from`, `graph_edges_to`, `graph_bfs`, `graph_transitive_closure`) return heap-allocated `*mut c_char` JSON strings that the caller must release with `graph_free_str`. Depends on `crate::graph`, `crate::loader`, `crate::types`.

### `src/guile_sys.rs`

Provides the Rust-side view of libguile's C API, gated to `#[cfg(target_arch = "x86_64")]`. Defines `type Scm = usize` and the four SCM immediate constants (`SCM_BOOL_F`, `SCM_EOL`, `SCM_BOOL_T`, `SCM_UNSPECIFIED`) derived from libguile header macros (`SCM_MAKIFLAG_BITS(n) = (n << 8) + 4`). Declares `unsafe extern "C"` bindings for `scm_cons`, `scm_from_utf8_string`, `scm_from_utf8_symbol`, `scm_from_double`, `scm_from_uint32`, `scm_from_bool`. Exposes convenience wrappers `scm_str`, `scm_sym`, `scm_u32`, `scm_f64`, `scm_opt_str`, `scm_list_from_iter`, `scm_alist` that build Scheme values by calling through to those C functions.

### `src/ffi_scm.rs`

Native-SCM FFI layer, compiled only for `x86_64-apple-darwin`. Functions parallel those in `ffi.rs` but return `Scm` (= `usize`) instead of `*mut c_char`. The Guile caller binds them with `#:return-type '*` and passes results through `pointer->scm` to reinterpret the `uintptr_t` as an SCM value — safe because the Rust code built the value using libguile's own C API within a Guile context. Helper functions `node_to_alist` and `stmt_to_alist` convert Rust graph types to Scheme alists with symbol keys. Additional exports: `graph_all_ids_scm` (full sorted node ID list), `graph_canonicalize_scm` (ID normalization).

### `bohemia_graph.py`

Two-level Python wrapper. The private `_Lib` class declares all C ABI signatures with `ctypes` and wraps each call; `_consume_str` reads a returned `c_void_p` as a UTF-8 string then calls `graph_free_str` to release the memory. The public `BohemiaGraph` class provides a Pythonic API with a context manager, `load()`, and five query methods (`get`, `describe`, `edges_from`, `edges_to`, `bfs`, `transitive_closure`) that JSON-decode the `_Lib` responses. At module import, it attempts `from bohemia_graph_native import BohemiaGraph` and silently replaces the ctypes class with the PyO3 native class if the extension is installed — so callers always import `BohemiaGraph` from the same path regardless of which backend is active. Library discovery searches `target/release`, `target/x86_64-apple-darwin/release`, `target/aarch64-apple-darwin/release`, `target/debug`, then `.`, relative to both the script's directory and CWD.

> **Note:** `BohemiaGraph.load()` accepts path arguments for API compatibility with the C ABI signature, but they are silently ignored by the Rust side — the graph data is embedded at compile time.

### `python/bohemia_graph_native/__init__.py`

One-line re-export: `from ._bohemia_graph_native import BohemiaGraph`. The actual `_bohemia_graph_native` extension module is built by `maturin` using the `python-native` feature, which activates `src/python.rs`.

## Tests

There are no test files in this repository. The project is a learning exercise / technology demonstration; correctness is validated manually by running the CLI (`cargo run`), Guile demo (`guile query.scm`), and Python demo (`python query.py`) and inspecting their output against known graph properties (713 nodes, expected edges from Holmes, etc.).
